# Sprint 1 "Sticky Retention" — Habit tracking, task completion, email nudges, weekly review

This sprint adds four backend agents (HabitAgent, ReviewAgent, extended PlannerAgent, proactive email sender) and rewrites the frontend HabitsDrawer against real API data. The design follows the existing agent-per-domain pattern with Gemini-based intent classification, Firestore persistence via repositories, and MCP tool calls for Google services. The proactive scheduler now falls back to Gmail when a user is offline, respecting notification preferences.

**Watch for:** Habit checkin and delete routes have no ownership verification — any authenticated user can operate on any other user's habit by ID (confirmed). The `record_completion` streak logic is purely additive with no idempotency guard, so double-tapping the check-in button (or retrying a failed network call) increments the streak twice (confirmed). The proactive email sender catches all credential errors into a single silent `False` return, so a revoked refresh token produces indefinite silent failures with no alerting (likely).

## High-level view

The HabitAgent classifies intent via Gemini JSON mode and delegates to HabitRepository. Streak counting is append-only: `record_completion` increments a counter and pushes a history entry, but never validates whether today's completion already exists. The streak number drifts from reality under concurrent or retried requests.

The checkin and delete REST routes (`POST /api/habits/checkin`, `DELETE /api/habits/{habit_id}`) verify the caller holds a valid Google token but never compare the authenticated user's ID against the habit's `user_id`. A user who knows another user's habit document ID can check in or delete it.

The proactive scheduler's email fallback constructs Gmail API credentials from stored refresh tokens and sends via `users().messages().send()`. `_send_nudge_email` is fire-and-forget — credential refresh failures, quota exhaustion, or revoked tokens all fall into `except Exception` with no structured retry or alerting.

The weekly review agent passes `days_back: 7` to the MCP calendar call, but no other call site in the codebase uses that parameter — if the MCP server ignores it, the review silently generates from an empty event list.

The frontend HabitsDrawer fetches from `/api/habits`, renders streak rings via SVG, and fires check-ins optimistically, but silently swallows all API errors with empty `catch {}` blocks.

## Habit ownership gap in REST routes

The `checkin_habit` handler calls `verify_google_token(body.auth_token)` but discards the returned user dict — it never extracts `user["sub"]` to compare against `habit.user_id`. The same pattern appears in `delete_habit`. Contrast with `get_habits`, which correctly queries by the authenticated user's ID.

```python
# checkin_habit (main.py ~L624)
await verify_google_token(body.auth_token)       # authenticates
habit = await HabitRepository.get_by_id(body.habit_id)  # fetches any habit
await HabitRepository.record_completion(body.habit_id)   # mutates without ownership check
```

Any authenticated user can check in or delete any habit by guessing the document ID. The fix is to extract `user["sub"]` and compare it to `habit.user_id` before proceeding.

## Streak idempotency and consistency

`HabitRepository.record_completion` reads the document, increments `streak` by 1, appends `{"completed_at": now.isoformat()}` to `history`, and writes back. No guard against multiple completions on the same day. The frontend's check-in button has no debounce beyond the 800ms animation timer. A double-tap or network retry inflates the streak.

The HabitAgent's `_check_in` method compounds this: it returns `new_streak = habit.streak + 1` based on the in-memory object *before* the Firestore update completes, so the response streak is stale-plus-one regardless of what actually persisted if there was a race.

A same-day check (comparing `last_completed` truncated to date against today) in `record_completion` would make this idempotent.

## Proactive email credential handling

`_send_nudge_email` instantiates `google.oauth2.credentials.Credentials` with stored tokens and passes this to `build("gmail", "v1", ...)`. If the refresh token is revoked (user removed app access), the call throws, is caught by `except Exception`, and returns `False`.

For a background scheduler running every few minutes across many users, a revoked token produces an indefinite stream of silent failures — the user never gets notified that their email path is broken. Logging the exception class at `warning` level and potentially marking the user's email path as broken in their preferences would improve operability.

## Weekly review data quality

`generate_weekly_review` passes `days_back: 7` to the MCP `list_events` call, but only `days_ahead` appears in other call sites in this codebase. If the MCP server ignores `days_back`, the review generates from an empty event list and falls into the "limited data" path of the Gemini prompt silently.

The `tasks` filtering uses `t.get("status") in ("completed", "done")` — Google Tasks API uses `"completed"` exclusively; `"done"` is dead code (never matches).
