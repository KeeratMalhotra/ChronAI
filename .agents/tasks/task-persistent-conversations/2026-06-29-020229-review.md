# Persistent conversation storage via Firestore subcollection

The change adds durable chat history so conversations survive page refreshes. Messages are persisted to a Firestore subcollection (`users/{uid}/messages`) after each WebSocket exchange, and loaded back on reconnect via both the WebSocket handler (for orchestrator context) and a REST endpoint (for the frontend thread). The frontend fetches history on WebSocket open, maps it into the existing message state, and shows a loading indicator during the fetch.

**Watch for:** Auth token passed as a URL query parameter to the history endpoint leaks credentials into server access logs and browser history (confirmed). The `_persist_exchange` helper assumes the last two entries in `conversation_history` are always a user/assistant pair, but multi-agent or error paths may violate that assumption (likely). No deduplication exists between the WebSocket-loaded context and the REST-fetched frontend history, meaning the same messages could appear twice if timing aligns (possible).

## High-level view

The storage layer uses a per-user subcollection (`users/{uid}/messages`) with timestamp-descending queries capped at 50 messages for the orchestrator's context budget.

Persistence hooks into the WebSocket handler via `_persist_exchange`, a best-effort function that extracts the last two history entries after the orchestrator mutates `conversation_history` in place. This works for the happy path but is fragile when the orchestrator doesn't append exactly one user + one assistant message — error fallbacks and multi-agent dispatch can both violate that assumption.

The REST endpoint (`GET /api/conversations/history`) accepts the OAuth token as a query parameter, following the project's existing pattern. This means credentials appear in URLs — acceptable for development but a liability in production where access logs capture full query strings.

The frontend loads history on WebSocket `open`, prepends it to local state, and exposes a `loadingHistory` flag. There's a merge hazard: if a user sends a message before the history fetch completes, the prepend pushes their message below older history, breaking chronological order.

<details>
<summary>Issues (5)</summary>

1. **Auth token in query string** — The `/api/conversations/history` endpoint accepts `auth_token` as a query param, leaking credentials into server logs and browser history. Accept it via the `Authorization` header only, or at minimum ensure the query-param path isn't the one the frontend uses for the primary flow.
2. **`_persist_exchange` pair assumption** — The helper blindly takes `conversation_history[-2:]` expecting user then assistant. If the orchestrator appends an error, a clarification prompt, or skips the user entry (multi-agent fan-out), it will persist wrong content or silently drop messages. Add role checks and index validation before writing.
3. **History prepend race** — If the user sends a message before `fetchChatHistory` resolves, `setMessages((prev) => [...mapped, ...prev])` inserts history above the already-sent message. Gate the composer (or buffer sends) until history is loaded.
4. **No deduplication on reconnect** — WebSocket disconnect + reconnect within the same component mount re-fires the `open` handler, prepending the same history again without clearing existing messages.
5. **Unbounded subcollection growth** — Messages accumulate indefinitely per user. The read query is capped at 50, but Firestore charges per stored document. Add a TTL policy or periodic pruning (similar to `NotificationRepository.prune`).

## Write-side deduplication gap

The `message_id` field on `ChatMessage` is stored but never checked on write. If `_persist_exchange` runs twice for the same turn (e.g., a retry after a transient Firestore error that actually succeeded), duplicate messages accumulate in the subcollection. At 50-message read cap this is low-severity, but a conditional write keyed on `message_id` would prevent drift between the persisted history and what the user actually said.

## Persistence timing and the `_persist_exchange` helper

```python
async def _persist_exchange(user_id, conversation_history):
    if not user_id or len(conversation_history) < 2:
        return
    user_msg = conversation_history[-2]
    assistant_msg = conversation_history[-1]
    if user_msg.get("role") == "user":
        await MessageRepository.save_message(...)
    if assistant_msg.get("role") == "assistant":
        await MessageRepository.save_message(...)
```

Failure modes:

- If the orchestrator hits an exception mid-generation and returns a fallback without appending to history, `_persist_exchange` persists stale entries from the previous turn.
- Multi-agent dispatch (`multi_intent`) concatenates multiple agent outputs into a single assistant message; the timing of when that concatenation happens vs when `_persist_exchange` reads the list needs verification.
- The role guards (`if user_msg.get("role") == "user"`) prevent writing garbage, but silently dropping messages means the persisted history diverges from what the user saw in the session — fail-silent rather than fail-visible.

## Auth token as query parameter

The `fetchChatHistory` call in the frontend constructs:

```typescript
`/api/conversations/history?auth_token=${encodeURIComponent(authToken)}&limit=${limit}`
```

The backend endpoint already accepts an `Authorization` header via `_extract_token`, so the frontend could switch to header-based auth with no backend change. With the query param approach, the OAuth access token appears in server access logs (nginx, cloud load balancer) and any intermediate proxy logs.

## Frontend history merge and the race window

In `useChatSocket`, the `open` handler fires `fetchChatHistory` and on completion does:

```typescript
setMessages((prev) => [...mapped, ...prev]);
```

The `prev` here could already contain a user message if the user typed fast. The composer's `disabled` prop is bound to `connection === "disconnected"`, which becomes false as soon as the WebSocket opens — *before* history finishes loading. So the composer is active while history is still in flight.

Fix: disable the composer until `loadingHistory` is false, or buffer outbound messages and insert them after history loads.

## What's not tested

No tests exercise `MessageRepository`, `_persist_exchange`, or the `/api/conversations/history` endpoint. The frontend has no test coverage for the history-loading path in `useChatSocket`. Given the race condition above, a unit test that simulates send-before-history-load would catch ordering bugs.
